# Derek Sorensen (Ethereum Foundation) Safely Snarkifying Ethereum: Formal Verification and Protocol

Hello. Hello, I'm Derek Sorensen. I'm part of the Ethereum Foundation research team. I'm on the protocol snarkification initiative, which is chiefly aimed at safely snarkifying the Ethereum protocol. So snarks, ZKVMs, this zero knowledge proof technology is central to Ethereum's roadmap to lean Ethereum, to scaling Ethereum, both at the execution and at the consensus layer. What a snark does in practice in this case is that it means that a client does not have to replay all of the transactions, but can instead accept a proof of correct execution so that the updated state is the result of correctly applying the transactions in a block to the previous state, which means that bugs in the cryptography and in the snarks that we're using can introduce... People can steal money, people can compromise the security of Ethereum protocol, which is, of course, a serious problem. So snarks are critical infrastructure in the long term for Ethereum Foundation, which means that on top of our normal testing and fuzzing and hardening techniques that we need to apply to everything that we ship, we need to treat snarks as requiring the highest assurance possible. And therefore, we should apply the methods that deliver high-assurance code, in this case, formal verification. Formal verification is a practice where you take a program and you take a formal description of what it means for that program to execute correctly. And then you prove with formal logic that your program satisfies that description of correctness. So this very simple example that I've included on the slide here is suppose you have a function and you've optimized it. So your function f, you've optimized it to a function f optimized. And the definition of correctness for this optimized function is that it behaves the same as your previous function, aside from performance. So kind of in an extensional sense, it behaves the same. So we can state this formally by saying that in this case, what's on the screen is a pure functional representation, but you can say for all inputs, for all states applying f optimized gives the same result as applying f. And that is a formal statement of correctness. That's what you mean. So if we can prove f optimized to be correct with regards to this description, we say that it's proved correct with regards to this formal specification, this description of correctness. And my slide has an asterisk there because there are some caveats that I won't say anything about now, but I will at the end of the presentation. Our work with snarks and ZKVMs as it applies to Ethereum is to obtain the highest possible level of assurance for ZKVMs. Our end goal is a bug-free ZKVM. And we're doing this work in three main tracks. I'll give a quick summary of each of these main tracks and then focus on the last one. So the first is we're doing a lot of work at the RISC-V level. All of the ZKVMs that are part of the ETH proofs initiative and that are candidates for the network of provers that will scale Ethereum implement RISC-V semantics at the ISA level. And so they're RISC-V ZKVMs. So what we need to do is we need to formally verify that the polynomial constraints implemented by these ZKVMs correctly encode the RISC-V ISA. And we have some really good tools to do that. First of all, the RISC-V ISA semantics are available in various theorem provers, including Lean, which is the one we use. And also we have developed tooling to extract the polynomial constraints into Lean. So to kind of to take what's in the Rust code and to put it into a place that we can reason about. Side note, by we, I mean about 30 or 40 collaborators in the whole ecosystem. So it's not just the Ethereum Foundation, but it's many, many collaborators throughout. Then at a level higher than that is the EVM level. Each of these ZKVMs will run the EVM in RISC-V. And so these are called guest programs. So we need to formally verify that the guest programs are correct implementations of the EVM, which seems like it might be a simpler task than the RISC-V semantics verification, but it has its subtleties. We have managed to prove, by we, I mean a collaborator, runtime verification, has managed to prove that Reth, the RISC-V code that Reth produces, is formally equivalent to a formal model of the EVM called KEVM, which is one of the first and oldest and most storied formal models. So that is a proof of correctness. There are some caveats here and a lot of work to be done. The state transition function was not formally verified in this case, and there are some questions about the boundary of verification. Last and finally is the cryptography track. So we have thus far cared about the implementations of virtual machines and the EVM and RISC-V, taking for granted that the underlying cryptography is sound. But the cryptography that produces a SNARK, a zero knowledge proof, is actually quite elaborate and complex, and it requires a lot of advanced mathematics and abstract algebra, including finite fields, polynomials over those finite fields that have elaborate operations on top of those. And so we need some way of evaluating these ZKVMs on the basis of their cryptography to ensure that the cryptography itself is sound. I don't know about you, but when I interact with kind of real-world protocols, sometimes I'm tempted to take the cryptographic guarantees for granted.
