Hello everyone, it's great to be here. Thanks for having me. I'm Jihoon. I do research engineering at the Ethereum Foundation. Fossil is the consensus layer headliner for Hydera, which is the upgrade after Gollamsterdam. In this talk, I'm gonna walk you through the past, present, and future of fossil, how we got here, how it works, how it can evolve and work with other features, such as frame transaction and blob streaming. The intuition behind inclusion list was simple. Let's make a list of transactions and force builders to include them. The idea was simple, but it took more than two years to reach the design we have today. This is not an exhaustive list, but let me briefly introduce the design space and the key decisions we've made. The first question is, who builds inclusion list? Should it be a proposal or should it be a committee? The main consideration here was, how easy it is to compromise. If it relies on a single actor, it's much easier to bribe or collude with the next slot proposal. Then there's conditionality. Should inclusion list be conditional or unconditional? Making inclusion list unconditional makes them valuable. They can be used for other purposes, such as pre-confirmations or as a vehicle for MAV. It also effectively increases the block gas limit as side effect. However, with conditional inclusion list, builders can stuff their blocks to ignore ILs, but the cost of that censorship increases exponentially thanks to EIP-1559. So fossil went with conditional. So we have a committee and the lists are conditional, but where do they go? Putting them in a block seems straightforward, but there's a problem. Producing an inclusion list is free, so a malicious includer can exploit this to get free data availability. They can just stuff some random data they want to make on-chain. And that's why fossil keeps inclusion lists out of the block and propagates them over the peer-to-peer network. We did some research on incentive mechanisms for includers. There are some technicalities, but the main idea is routing a portion of transaction fee to includers. But it does open the door to a lot of complexity, which we don't believe is justified. Let me introduce that ongoing research later in this talk. Builders listen to the ILs and make sure their payloads include all the transactions from them. They can order transactions however they want, and during network congestion, if their block is full, they can favor their preferred transactions over IL transactions. But they should include IL transactions as long as they can. Attestors also listen to ILs and check if the payload includes all IL transactions. If it does, the payload, they vote for the block. Let's start with frame transaction. A frame transaction is a new transaction type that decomposes a transaction into abstract units called frames. And it also allows each frame to be more generic, so this structure gives us expressiveness that legacy transactions don't have. In a transaction, you can find different set of computation. For example, verification where you check nonce, balance, and signature, execution where you perform the operations you want. If you are using account abstraction, like ERC-4337, you might have operations at the end such as gas reference to the paymaster. You can see each group of operations on the left maps to a corresponding frame on the right. This is abstracted and simplified code for the sake of explanation, so the actual code will look a little different, but I hope this gives you the intuition. Let's walk through a few examples. Imagine you want to swap tokens. Today you need to make two transactions, one for approval and one for the swap. But with frame transaction, you can represent these as separate frames. You verify first, then approve in the next frame, and swap in the last frame in one transaction. Let's look at another case. Say you want to use a sponsored transaction. First, you need a paymaster who has staked it in the entry point contract. The paymaster has to deposit in a fund to sponsor transactions, and that's actually the first transaction. And a user submits a user op to a bundler. The bundler checks if the user op is valid and wraps it into a transaction. Similarly, these stages can be expressed as frames. Each frame can verify that the paymaster approved, pay the fee, and refund the unused gas to the paymaster again in one transaction, but without going through off-protocol actors. This is good, but it comes at the cost. Frame transactions turn validation into code execution. What I mean by that is, unlike legacy transactions, frame transactions have to execute verify frames for validation. It is powerful because it gives us some sort of programmable validation, but this calls for some mempool rules to protect nodes from resource-intensive validations, such as verify gas limit, accepting only one pending frame transaction per sender, allowing only context-independent opcodes, and so on and so forth. But I do want to point out that this is not new. ERC-4337 has its own validation rules, and it inspires frame transaction rules. However, ERC-4337 rules are a bit complex, so there is ongoing research on finding rules that are less complex but suitable for frame transaction. For frame transactions accepted into the mempool, fossil would add extra constraints on which ones can go into ILs. This is to make sure the work for builders and attesters are bounded and manageable. It does narrow things down, but we are aiming to find parameters that can cover most of the use cases. So to summarize, frame transaction empowers users to interact with the protocol directly without going through intermediaries. So you can use frame transaction for native account
