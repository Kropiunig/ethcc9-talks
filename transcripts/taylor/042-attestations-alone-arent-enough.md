# Attestations Alone Aren’t Enough
**Peter Us** (Oasis Network - Founding Engineer) --- Taylor Stage

Hello, I am Peter Uss from the Oasis Protocol Foundation. Welcome to my talk. The first slide is not working. The talk is titled Attestations are not enough. Let's talk about attestations and TEE and what it takes to build practical applications using this technology. So, as a user, once you visit a project that is using TEE nowadays as their security primitive, I usually see a bunch of green checkmarks that say something like verified, trusted, attested even, and then a bunch of raw data that nobody really actually understands or reads them. And unless your users are security researchers, I think this is more of a security theater than bringing any actual security to apps using TEEs. So as a high level, what the TEE gives you, you get three things basically. You get isolated execution, which means that your code is run privately in a part of the CPU that even the operator cannot see. You get per CPU root of trust. These are basically some cryptographic keys that are baked into the CPUs at the time of production. And the third thing is the remote attestations. And this is a quite powerful concept, which allows us to prove to a third party that we are running the code that actually we are saying that we're running without having to trust the operator directly. And all of this is nice and necessary, but it's not enough in practice to actually build applications that can be considered trustless in a way. So let's focus on Intel AGX attestations for this example. So we got this, what is called attestation quote. What can a user do with it? So they need to parse it. They need to extract certain fields out of it. There's a like 100-page PDF provided by Intel, which describes how to do this stuff. You need to validate certificate chains, that these things actually came from Intel. You need to interpret different fields inside the quote. You should check freshness because a quote might be old and if it's old, maybe the code changed since then. Maybe the architecture platform that's running it is not trusted anymore, et cetera. And ideally, we should be doing all of this continuously. So in essence, what the attestation provides us, it's three things. It provides us that a specific hash of a code was run inside a specific enclave. It also tells us that that enclave or that hardware is deemed secure, whatever that means by the hardware provider, and that this happened at some moment when the quote was produced. And on the other side, what users assume or maybe even what users kind of are led to be assumed or like kind of what projects using these try to tell users that this automatically means that the application is safe, that it has no bugs, that it's actually running the version that it says it's running, and similar. So, and now even if we like get attestations, we verify the quotes, we do everything correctly, we do this continuously, we can still get examples like this. So this is a valid attestation that's produced by Intel, but then you can check a bit more in depth into it. For example, this specific attestation was done on the third generation Xeon CPUs and those CPUs actually have very well-known and very practical attacks that are basically able to completely break the machine and extract the hardware keys that were baked into them. But the nice thing is that Intel considers these features as out of scope. They see it as a feature, not as a bug. Because their threat model explicitly states that requirement of a physical access of the machine is out of their threat model. So what this shows us is that we need actual policies on these attestations that go beyond just does the hardware provider say this is trusted or not? Because your threat model might want to involve also the physical requirement of presence being a part of your trust model. Maybe a bit of a side note. As I mentioned before, the attestation only proves you that a specific hash of a code was running inside the secure enclave and this is meaningless unless you make your source code public and also reproducible. Because everything else otherwise doesn't prove you anything because if you can't independently compile the code, get the hash of the code, and ensure it matches the attestation, the security model collapses. Luckily, this is solved in 2026. Just developers need to be a bit more mindful. It's a bit more tedious sometimes, but it is a solved problem. Now let's say we have some TE application that's running somewhere in the cloud. You get the public code of the application. You compile it. You verify it. You check it has no bugs. Maybe it's even reviewed. Maybe it has security audits done. And everything checks out. The attestations pass. Everything matches. Well, that might not be enough. Because maybe you checked the previous version of the code and maybe it had a bug where it leaked the private key out. Or maybe it had a debug statement that was left in. And this example is a toy example, but actually showcases that it's not enough to just verify the current version of the code, but you need to be able to verify all the past versions of the code as well. Because if the upgrade path of the application isn't transparent, again, the trust model of the application collapses. So maybe just as a short recap, what attestation doesn't give you. It doesn't give you any trusted computing-based governance beyond what the hardware provider says and he deems trustful. It doesn't tell you anything about previous versions of the code. It doesn't enforce freshness. You need to manually check that the quote was produced recently. And also it tells you nothing about the operator. As we saw, maybe some CPUs have attacks that actually require trusted operators, not just the trusted code or the CPU. And ideally, in an ideal world, we'll be doing all of this all the time. So we would be verifying attestations. We would be enforcing freshness. We would prevent rollback. I won't talk about this much, but this is an important thing because it makes state rollback lead to a bunch of other attacks. We should be able to apply custom policies that go beyond what the provider says it's valid. We should be able to track upgrades to the applications and ideally also know which operator is running the code. And this sounds like a good job for a blockchain system. So this is what the Oasis Network has been doing for the last five years. You can look at the Oasis Network as a continuous attestation verification system. It supports on-chain policy definitions and enforcements. It's run by independent and economically incentivized operators. The code is open source and it's public. It's went through audits. There's bug bounties out. And the whole idea of this is that instead of putting the burden of verifying attestations to the users, let's have a agreed upon consensus from operators doing this stuff and the users can then use the consensus network state as a root of trust. So for the user, this means that now the raw attestations, he doesn't need to see them. The raw attestations now become the inputs for our network and the consensus decision if something is valid or not or approved becomes the output. And users now verify the Oasis on-chain state and don't even need to know or see the raw attestations. And this also gives us the ability to abstract the attestations from multiple providers in a unified model for the end users. So once we have this, like our Oasis network or some other network that we have some verified on-chain root of trust, we can start thinking on how we use this to build trustless applications that actually do something useful. So at minimum, we need like on-chain registry of TEE applications, which each app should have an ID, an owner, the currently approved code measurement. We should track updates and ideally also all enclave starts on-chain so the user can see the history. They can see what was the previous versions of the code. They can actually see how many instances of the application is running currently. And this should be done periodically. So about two years ago, I think, we launched the Oasis Ruffle, which stands for Runtime Off-Chain Compute Logic. And this, in essence, allows arbitrary containerized applications to be run on the Oasis operators. And Oasis Network...


Oasis Network here is just used as the attestation and the trust layer. So this can be completely off-chain applications. It can interact with other chains. The Oasis network is here mostly used as a security mechanism. Now, once we have this on-chain root of trust, we can do more useful things inside the apps that we call Ruffle apps. So, for example, instead of, again, putting the burden of the user to verify the Ruffle apps that they are attested and running via the on-chain state, we can put that inside the code, inside of the every Ruffle application. So every application that's running on Ruffle in Oasis actually needs to contain a startup code, which does exactly this thing. It fetches the on-chain state and it actually verifies itself. So it can check, is my code the one that's stated on the on-chain state? Am I running on an operator that's in good standing that I'm allowed to run on? And similar policies like this. And similarly, if we have users interacting with these applications, they can follow the exact same path. What they need is a trusted access to the on-chain state. This can be done by light clients. They fetch the registry of the registered apps. They get the code hashes there. They can see what policies are defined. These are all public. And they just compare what they see with what's on the on-chain state. And again, like here, no attestations are ever exposed to the clients. So we've been building applications on top of this for about the last two years. And we've been, we noticed or we found some interesting patterns, like primitives that then enable more powerful applications to be built on top of this. I'll go over a few examples now quickly. So first thing is, we call this attested TLS. It's basically how can you, if you run a server inside the TE, how can you establish a TLS connection directly with that server without relying on any third-party certificate authorities or something like that? And the idea is quite similar to the previous ones. So the TE application on startup, it generates public and private key pair for TLS. It publishes the public key on chain. And there's like, it's important to know that there's a part of the on-chain data, which only the Ruffle app itself can update. So that can be trusted that it came from the application itself. And then as an outside client, you connect to the server, you get a certificate, you again use something like a light client to get the Oasis on-chain state. And you can verify that the certificate matches to the public key that's published on chain. So you get the full TLS directly inside the enclave without relying on any third-party custom authorities or anything similar. Another very useful feature is what we call encumbered wallets. This was actually coined by the research paper made by Cornell Tech, by the group of Ari Juce and others. It's called the liquid fraction paper. And in essence, this is, these are basically keys that live inside enclaves and TE applications. They were generated there, never exposed. And then they defined access to these keys via on-chain policies. And not actually access to the keys directly, but access to signatures made by these keys. So this enables a bunch of useful use cases. The paper itself listed a lot of them. But one cool example is Take My Ape. That's actually what they built as a proof of concept of their paper. And what they did is, they have an Ethereum account that was generated inside the Oasis Raffle TE. That account actually owns a Bored Ape on the Ethereum mainnet network. And then they built an on-chain bidding process where users can build to get access to that key for a limited period. So with that, you can, I don't know, pay $5, $10, get access to the Ape for five minutes. You can use it to produce signatures that actually prove that you own that Ape. And then you get, I don't know, you can get invited into the Discord, which is only for the owners of the Bored Ape Yacht Club and things like that. And the important thing is here that the user now gets the access to the private key. There's actually on-chain code that enforces only signatures for proving the owning of the key. You cannot generate a signature that would transfer the Ape to another account. Another cool thing is FlexFolds. This is what our team is working on. It's basically our take into taking the liquid fraction paper foundations into production with a bit more focus on DeFi. And the pattern is quite similar. We can have keys for a bunch of different chains that are controlled by TEs. There is a policy engine which users can use to define custom policies like time-based spending, spending limits, token whitelist. There is DEX support integrated into it. And here again, Oasis Network is only used as a security for this software. This is not limited to Oasis Network in any way. The idea is that you interact with other chains through this. One thing that's common to all of the use cases that I mentioned is that we're generating keys inside TEs. So if you're using plain TEs, you'll get some hardware-provided keys there. But those keys are always tied to the code measurement. Basically, if you update the code that's running in a TE, you get new keys. This doesn't really work in practice because for building useful applications, we need upgrades. And for that, our network also provides a decentralized key manager where keys are bound to the app ID. So we call this like consensus-based key provisioning. As I talked before, every role application registers on-chain. There is a committee of TEs that we call key manager. They run a decentralized protocol for sharing keys. And again, these TEs use the Oasis network as a root of trust. Root of trust who gets access to which specific keys. And now these key managers can provision the exact same pair of keys to the new version of the TE. And yeah, let's just to wrap up, let's go over the takeaways.
