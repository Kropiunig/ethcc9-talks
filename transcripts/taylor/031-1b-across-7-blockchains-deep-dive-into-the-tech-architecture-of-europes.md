# $1B across 7 blockchains: deep dive into the tech architecture of Europe’s
**Nicolas Assouad** (Spiko) --- Taylor Stage

Hello everyone. I'm really happy to be here today at ICC. My name is Nicolas and I'm a software engineer at Spyco. Today I will talk about how we manage over one billion dollars of real world assets on seven different blockchains. First of all, what is Spyco? Spyco is a new regulated investment firm licensed by the AMF in France. We managed tokenized financial products on public blockchains as smart contracts are open source and the audit reports are available online. Basically, the financial products that are tokenizing our flagship today is a money market fund product, it's T-bills. As of today, we have practically $1.5 billion assets under management, which makes us the third biggest actor of tokenized T-bills globally. I told you that we are live on seven blockchains. Here they are. We are live on five EVM compatible blockchains, namely Ethereum, Polygon, Arbitrum, Base and Etalink. But we also live on StarkNet and we live on Stellar. We plan to be live on many more blockchains and there are several coming up this year. But this setup brings us a very technical engineering challenge. How we can safely operate on seven blockchains in a consistent manner without relying too much on blockchain-specific implementation. To answer this question, I need to talk to you a little bit about our goal and our vision. For us, blockchain, at first, it's a concept more than an implementation or a technology. We want to rely on core blockchain property, such as transparency, auditability, security and traceability. That's the main principle on which we want to build Spyco and that we rely on this principle. So blockchain is really our core technology. Everything in Spyco is derived from the blockchain. The blockchain is really our single source of truth. In order to be able to design such an infrastructure, we need to have a standard blockchain agnostic smart contract architecture, which has the same separation of concerns, the same role, the same workflow, the same lifecycle. That's for the on-chain part, but there is also the off-chain software component that also needs to be blockchain agnostic and generic. The idea at the end is to be able to add a new network, not in months, not in weeks, but in days. It should be as simple as knowing and implementing smart contracts in a given technology for a given new blockchain and implementing a bunch of software components for all our often interactions. Here it is, our smart contract architecture. As you can see, we aim it to be quite simple. It's composed of four main smart contracts. The most important one is the token smart contract. The token smart contract represents one financial product on our platform. So we have one smart contract token deployed per financial product. For example, I talked about money market fund. We have a euro money market fund, which is called UTBL. It's a smart contract, a token smart contract. And we have a dollar money market fund and it's also a token smart contract deployed on-chain per network. If you're familiar with the EVM ecosystem, it can be seen, and it's actually the implementation for the EVM implementation, as an ERC20 permission token. But really, that's the EVM implementation and we want to see it as a generic token implementation. So there is N instance of the token smart contract. And then we have the three other smart contracts, the minter, the redemption and the permission manager smart contract, which are singletons. So there is only one instance of each of these smart contracts per network. And they are the smart contracts that handle the security and the operation of the fund on-chain. So these smart contracts ensure us that we are operating the fund properly and securely on-chain. And handling each of them are handling their own flow, so the minting flow, the redemption flow and the permission flow. So we will look a little bit deeper on each of these smart contracts to give you an idea of how it works and what guarantees it gives us. So to begin with, the permission manager contract. It's one of the most important, actually. It gives us, it's really the central permission authority. It can be seen as an access control manager for all the other smart contracts, but also addresses, as I said, our token are permission token. So we need to whitelist all the investors that have wallets and want to invest on Spyco. So it's not a complete list of all the roles that can be granted in our setup, but some of them here. And as you can see, there are several personas in our setup. You have the multisig, some smart contracts, the relay wallet and the investor wallet. So the multisig, it's really our team operational wallet. So that's really the admin. It's a human operated wallet, so it's not programmable. And it's the wallet that can grant the sensitive role to the other part of the system. It can grant the role for the smart contracts, different smart contracts, which have dedicated role, which allows them to efficiently go through the different flows, the redemption flows and the minting flows. But there are also the relay wallet. The relay wallet, they are operational wallets, but which are managed programmatically by Spyco app. So this wallet, the relay wallet, are initiating all the operations of our fund. For example, they are dealing with the whitelisting operation. So they have the whitelister role, but also they are initiating all the operations that will then interact with the other smart contract, for example, with the mint initiator role. And then we have the investor wallet, of course, which have the whitelisted role. It's quite important to keep in mind because our setup is hybrid. So we handle at the same time custodial wallet. So investors that delegate the custody of their wallet to us and self-custodial. So our system should work efficiently with both cases. So the minter smart contract. So to follow all this role setup, let's have a look at the minter smart contract, which is using this permission setup to effectively control how we can mint tokens in our system. So the minter smart contract, it should be seen as our safeguard again against infinite mint problem, bugs or attacks. So internally, the minter smart contract holds a daily limit for each token. So basically each day you can only mint a fixed amount of token automatically. And then it needs to go through a two phase approval. So basically our operational wallet, as we've seen before, which have the mint initiator role, can interact with the minter smart contract. It will try to initiate.


init a mint, for example, when we've received money in the fund and we want to issue tokens that are related to this subscription to the fund. And the minter looks at its internal state and looks if it's under the daily limit. If it's under the limit, it passes through and it invokes directly the mint function of the token, and that's it. But if it's over the daily limit, it's blocked and it's logged inside the internal state of the minter smart contract. Then it needs a human intervention from the multisig that is the only one which has the mint approval role, which is really the safeguard of all this setup, and which is the only one who can approve or cancel a mint operation that has been blocked. How do we know when a mint is blocked or not? The minter smart contract, like all our smart contracts, have a standard set of blockchain events that can be logged inside the blockchain. It's events in EVM and it's a standard component of all the blockchains that we are operating on. And so we monitor for the mint initiated events, there is a mint blocked event, and with all these events we know when we should intervene with a human operation to unblock the situation. The other piece of the puzzle is the redemption contract. So the redemption contract, it's quite similar to the minter contract, but it serves another purpose to help us to mimic how our fund is working, because at the end we are working with real-world assets, so we have real-world obligations and constraints. And basically we are working with wallets of investors that are custodial that we own, and wallets that are self-custodial that we don't own. So we want an experience that is fluid and simple for investors, so we want them to be able to redeem with one operation and then they don't care when the fund is really doing and executing the redemption depending on how the fund is working. For us, for example, nowadays we have one fund execution per day, so we have one time each day where we can execute the redemption. So to tackle this problem, we have also a two-phase workflow. First of all, the investor will transfer its token to the redemption contract, and at this phase we can see the redemption contract as some kind of vault, segregation account, where the fund are segregated and waiting to be redeemed effectively. And so it's stored in the internal state of the redemption contract, and it's waiting for the operator to execute it effectively. And then when we effectively, with the fund, process the different redemptions and to leave the blockchain to go to the real world, our operational wallet call and interact with the redemption contract to execute the redemption effectively with the proper ID to call the burn function of the token. So there is really two flows, two main flows, the minting flows and the redemption flows, and these contracts are the only ones that can interact with the token's sensitive functions directly, namely the mint and the burn function. So that's for the on-chain architecture. What about the off-chain story? So for our off-chain architecture, we wanted to follow the same philosophy. So the idea really was to implement a stack and an application that is the most stand-out Web3 application that we could think of. So the components are what every blockchain application should have, namely the relay infrastructure and indexing infrastructure. But what we wanted to focus on is that, as I said, blockchain is a core infrastructure. We rely entirely on the blockchain. So it's really a core business technology for us. And so we need to own all the core logic of the interaction with the blockchain. So we wanted to limit as much as possible our exposition to third-party providers. So basically, we have two third-party providers that are really thin layer. The first one is our wallet provider, which nowadays is defense, and which can be seen really as a key management system. Basically, it's our blockchain primitive provider. We call it to sign transactions, and that's it. So that's really a signing function provider. And the other third-party provider is the RPC provider. It's mainly a node of the network, and that's it. All the rest is in-house. The relay and the indexer have been developed in-house to own really the broadcasting logic, which is different from every blockchain with different ways to ensure unicity, to ensure repeatability, etc., and to own completely the indexing part, which also is very blockchain-specific. Also, the idea was to be able to have as much traceability as possible. So every time we can do it, we try to match all the events we get back from the blockchain to the transactions we're sending. So that's making really a loop that allows us to know perfectly everything that is going on-chain and everything that has been initiated by us very easily. So SpyCo app, the SpyCo system broadcasts the transactions through the relayer, so all the blockchain is abstracted from all the other parts of SpyCo system. And the relayer talks to the blockchain and sends the transactions, and then the events come back through the indexer and then hydrate all the SpyCo service. From the point of view of SpyCo service, it's really only events that exist. The rest of the blockchain does not really exist. So to conclude, what we've learned when we implemented this cross-blockchain design is that really blockchain, it's a concept. You really don't want to rely on specific implementation or specific technology that change, the standard change, and the new blockchain that can pop every time. But if you rely on the concept of blockchain, you can redesign an application that can grow and scale through time to the current technology but to the next technology that are to come.
