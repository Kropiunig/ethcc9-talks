# How Ethereum Can Scale to Billions of Users
**Ed Felten** (Offchain Labs) --- Monroe Stage

Good morning, everyone. I'm Ed from the Offchain Labs team, developers of Arbitrum. And I'm here to talk about what we can do to scale Ethereum to a billion users. We talk about that a lot. Many people talk about the billion user goal. But what I hope I can offer this morning is some real talk about what we need to do to get there. What is it that we're building? And how can we build it in a way that will attract that billion users? And I wanna start with Paul Graham's famous phrase in advice to startups, make something people want. What this means is having some humility about what it is that we are building or should be building in order to focus on the needs of users and developers for our platforms rather than building for ourselves or building something that we think is cool. So what is it that people want? What is it that we can provide that really has mass appeal and value to users all around the world? And I think at the end of the day, what it is is I would call the programmable economy. And the idea is to take the economy that people are already engaged with and in and to make it programmable in the same way that the internet took communication and made it programmable. We wanna take the economy and make it programmable. That ultimately is, I think, what people want and it's the thing we can build that will be attractive and bring those users on board. It should feel like the familiar economy, only faster, cheaper, and with less friction. So I think the programmable economy has four main components that we need to provide. The first one is open and accountable code. That means not only that people can see what is the code that is implementing this, what is the code that's implementing applications, but also that there's some sense of accountability there so that people know how this can change, where it came from, and if there are changes that people don't like, they have some recourse. The second one is permissionless innovation. We want it to be the case that anyone who has a good idea and some reasonable capability or excellent AI can build a new product, a new idea, and roll it out at whatever scale the market will, that matches the market interest. The third one is scalable connectivity. If we, as we build this, we can't build a single monolithic system that will be able to serve all of these people directly. We need to figure out how to scale up the system while still retaining connectivity so that people who are connected in different places or assets in different places are able to interact in a relatively smooth way. And then the last one is integration with the economy. That is not to build a separate economy alongside the legacy one, but to integrate into the legacy economy in a way that makes it better, faster, cheaper, and easier to use. So how are we doing so far against these goals? Well, in open and accountable code, we are killing it. Not only is the code that provides our platforms open and accountable, but also, almost uniquely among technology platforms, the Ethereum stack and similar systems is able to provide and support open and accountable code for applications that run on top. People can inspect the code. They can know that the code that is running is the same code they inspected. And that is really a unique value that we can provide. And it's one reason why, at least, nerds are really attracted to this technology, but it can provide a lot of value to end users if we can make it practical. Second, permissionless innovation. There, I think we're also doing very well. As the tools grow, it becomes easier and easier for people to build systems and roll them out worldwide. Now, in the last two, I'd say, are works in progress. Scalable connectivity, we have interop across the Ethereum ecosystem and to some extent outside, but that interop is in many ways not ideal. And I'll talk more about this in a few minutes. But it's still a work in progress to get to the level of trustlessness and the level of friction, of low friction that we need to reach. And finally, integration with the economy. Again, a work in progress. I think the entry of large TradFi firms into the space and the growing integration between blockchain-based DeFi and TradFi platforms is really exciting. It's one of the things that we have to do to get to that billion users, but it's very much a work in progress and we need to continue to work on that. Okay, so how can we get there? What lessons can we take? It's important always to learn from others' success. And I think one of the others that we can best learn from is the internet, which went from this map in 1973, each oval being a computer, to this map about 25 years later. And now you wouldn't even try to put it on a diagram. So the internet is really the unique and best example of an open and permissionless system that was able to grow beyond a billion users and to do it quite quickly. So I think there's a lot we can learn from what the internet did. So lessons from the internet. First, building layers. The internet famously was built in layers. And that's one of the reasons why it was able to outcompete its monolithic competitors. So building in layers gives you a few advantages. First, each layer can focus on doing one thing and doing it very well. And each layer can rely on the layers below doing their things well, so you don't have to replicate that functionality in the same way that in the Ethereum stack, L1 provides a very high security, very reliable base on which to build everything. And layer twos can provide, can take advantage of that to provide additional features, higher throughput, and so on. Similarly, on the internet, you have this stack of layers. And as you go up the stack, what you find is each layer innovates more quickly and is willing to take a bit more risk in its innovation. And innovation increasingly happens as you move up through the layers. The internet was very successful with this. Ethereum has been very successful so far with this. And I don't see a reason to go away from it. Second lesson from the internet, make it programmable. This one we already have. Next lesson from the internet is connect everything. Now the internet does connect everything, but it connects everything in a particular way. And I think we can learn from how the internet connects everything and emulate that as we work to connect everything across the Ethereum stack. So I think interop that works, interop internet style, has a few characteristics that I wanna point to here. It's fast. It's standards-based. We're doing okay on these things. But I think the last three have important lessons for us. First, it's asynchronous. One component doesn't stop and wait for another component because at global scale, they can't really be synchronized and also run at maximum speed. So don't stop and wait. Next, it's fault tolerant, which means you don't assume that the other one will be available. You need an interop scheme which is able to deal with the other component being down, being non-responsive, being disconnected from the network somehow. And it's security conscious, meaning that one component doesn't automatically assume that the other component is secure. It's a try to interact, but protect yourself against the other component being compromised. And so as we build out interop across the Ethereum stack, I think it's important to be asynchronous, fault tolerant, and security conscious. And that will be the key to being able to scale it up. And that means that the friction will not be zero, just as on the internet, the friction from interop and operating across different sites and services is not zero, but we can drive that down. And in fact, in practice, you get the lowest friction by following this recipe. Lessons from the internet. Everyone chooses who to trust. We don't assume that there's a central body that decides what's trustworthy and what's not trustworthy. Each user can make that decision for themselves or delegate it to some entity that they trust. And finally, innovate to solve users' problems. That is, we should innovate not to solve our own problems as users, nor should we innovate to build something that because we think it's cool. We should figure out what it is that users want and need, and then direct our innovation towards those problems. And I wanna do a deep dive into an example of this, what I think is an excellent example from the recent history of Arbitrum. And that is a user problem of gas price variation. So I have a graph here that shows the price of the base fee, the gas price on Arbitrum 1 on October 10th of 2025, which some of you may remember was probably the busiest, highest market turbulence day in the history of our stack. And this is against UTC on the X-axis, the Arbitrum gas price. And you see that it peaks at around 41 Gwei. What that is, is about a 2000X increase in the price of doing a transaction. And of course, this happened at exactly the moment when users cared the most. And if you go to any consumer of any product and say, this thing you rely on, someday, unexpectedly, it's gonna cost 2000X more than it does today. And that will happen exactly on the day when you most want that thing. That is not a user-friendly situation. So we set out to attack this problem. Could we reduce gas price variation without putting in danger the things that the pricing algorithms are already protecting? And this is a hard problem. And it's a hard problem for a fundamental reason. That first of all, the demand for transactions for block space varies incredibly over time. And second, you have chain infrastructure, which has limited capacity. And so the conventional wisdom is that you have to increase the price.
