# Fast finality is essential for mass adoption
**Ellie Davidson** (Espresso Systems) --- Monroe Stage

Right? Hello. So I am Ellie. I am the chief product officer at Espresso. And today I'm going to be talking about why fast finality is a requirement for mass adoption, but particularly institutional adoption, because institutional adoption is quite the theme lately. But we'll get into it. So to start off with, I kind of want to give the framing. And this talk is a more kind of philosophical talk rather than perhaps a deeply technical talk. But the world itself, completely disregarding chains, runs on these sovereign systems. And some examples are, you know, you have governments, you have companies, you have individuals. And they each have these kind of weird dynamics between them. So, for example, you have governments. And governments interact with each other differently over history, over time. You know, companies can belong to multiple different governments. You have individuals. Individuals belong to companies. They belong to governments. They belong to all of these different systems. But an individual itself is a system. And what this results in is that you have, you fundamentally need to have trust between all of these systems in order to interact with between them. But this trust is very ambiguous. It's ever-changing. And it's really complex. And it's very difficult to understand what's happening. But we need to interoperate between these systems. And a lot of times in blockchain, we talk about interoperability, specifically in the form of bridging. But this is not unique to blockchain. Like, interoperability is actually a problem that we have, that has existed for a very long time. And the more efficiently we can interoperate between these systems, you know, the better that our world will work. And so trust-minimized base layers are the most effective way to interoperate between all of these systems. And a base layer is not something that is unique to blockchain. A lot of the times we think of Ethereum as being, you know, the trust-minimized base layer that we should all use. But there's many other kinds of base layers that actually are not on-chain at all. So, for example, you have certificate authorities. They are functionally very similar. You have someone that you trust at the top who is essentially operating as a base layer and the root of trust. And then everything else falls down from there. You have CSDs, for example, like DGCC, Euroclear, Clearstream. These are off-chain, but they operate as like this central root of trust so that you don't have to trust all of these individual counterparties. And then, you know, you also have things like identity, right? You sign in to an account with your Gmail account or with your Apple ID. And the idea is that you only need to trust just that one core component, and you don't need to trust all of this, like, complicated web of systems. So what a trust-minimized base layer does fundamentally is it simplifies this problem. So it collapses all of these many bilateral trust assumptions into kind of one multilateral trust assumption. And this is much easier for us to reason about. And this is a pattern that we have seen completely disregarding blockchain. We have seen that this is a workable pattern in the world in order for us to kind of navigate these very complex systems. And I'm going to argue in this talk that finality is like the primitive of all of this trust. And so that is why we need to focus on fast finality specifically because it is the key primitive that enables the sufficient interoperability. So we have our favorite base layer, Ethereum. You know, Ethereum itself is kind of this root of trust for its layer 2s. And this model works very well, arguably. Layer 2s can be highly customized. They can do a lot of their own thing, and then they inherit this really strong security from Ethereum. And when layer 2s want to bridge or interoperate between each other, you know, the most secure way to do it is to root that bridge in, you know, Ethereum finality itself. But Ethereum is not the only base layer. And it's not going to be the only base layer. So, for example, Espresso is also a base layer. But like I said, you also have base layers like, you know, these CSDs, things that are not associated with a blockchain whatsoever. And we kind of have this fundamental tension where we say, okay, we want to remove all of these trust assumptions. And so the most ideal pattern would be if we have one single base layer that we can all agree on. But that's obviously not how the world works. So, for example, the world does not have a single government. The world does not have a single body that governs all of the, you know, financial activity across the entire world. You naturally have these different jurisdictions. And this is a kind of a good analogy to what's happening on-chain. So Ethereum is a wonderful base layer, but it's not going to be the only base layer. And that's okay. What we want to do is kind of find a medium ground between having too many disparate systems, but also maintaining the sovereignty and the control that, you know, each one of these systems kind of needs. But we need interoperability between these systems. And this is where finality comes in. And I'm going to argue in this talk that we need real-time interoperability, which means that we need real-time finality. And I don't think that this concept is new to people. I don't think there's, you know, I don't think anybody out there would argue that, no, Ethereum should actually have worse finality times. I think everybody's fairly in agreement. But what I'm going to argue here is that it's actually a requirement for, like, massive institutional adoption rather than this kind of, like, nice to have. I think a lot of the times you're like, hey, well, you know, institutions are slow today. They have T plus one settlement. You know, Ethereum finality time is way better than that. So it's really not a requirement. And I think, one, we shouldn't compromise. Having real-time finality really is the most optimal architecture to interoperate between all of these things. And second of all, there's, like, very meaningful benefits and efficiencies that you get from real-time finality. So, yeah, like I said, the future is that you have real-time finality. And I'm going to define real-time as essentially sub-second. I would say, like, 500 milliseconds is kind of a good benchmark of what real-time finality is. So, well, why do we have poor finality today? Not just among chains. Like, I think a lot of people argue, actually, you don't have poor interoperability between, you know, L2s and Ethereum today. You have things like solvers and, you know, that give you very fast bridging. So we don't. And I'm going to argue that that's not quite true. But the reason that we have a lot of friction is because you have slow finality and slow proving times. So, as many people are aware, finality and proving times are not quite the same thing. They can be intertwined. You know, but finality itself is the point where, you know, something is irreversible. And how you prove the state on top of that can come in a variety of different ways depending on your use case. So, for example, Ethereum comes to finality after 15 minutes. And that is where, you know, it is irreversible and anybody can deterministically derive the state. Now, if you run your own full node of Ethereum, you can deterministically derive that state immediately. So, you know, your proving time is essentially a few milliseconds, whatever it is to execute that Ethereum block that you're following. If you do not want to run your full node, then maybe you need to wait on a ZK proof to be generated. And that's going to take quite a bit longer. So, proving times are a big part of interoperability, but, like, finality is really the fundamental primitive. And so I'd kind of put it in a pyramid where fast finality is the primitive and then proving times are kind of on top of that, but they're, like, use case dependent. But I think it's really interesting to look at the structure of interoperability here because, you know, I talk about fast finality, but in today's landscape, proving time is actually the bottleneck. Like, I am all for Ethereum should get faster finality. You know, Espresso has very fast finality, but finality time is actually not the bottleneck today at all. It's proof time. So let's look at the structure of an interoperability in a trustless way of an optimistic roll-up. I have these teeny tiny slivers to show the latency of, like, a sequencer inclusion, how long it takes for you to actually publish your data to Ethereum, how long it takes for Ethereum to come to finality. And then you have the time it takes to generate the fraud proof, which right now is 7 days. And that is just massively longer than anything else. So I think the number one thing that you can do right now to improve interoperability, not only among Ethereum, but just in general, is you need to reduce the proof time. And I think, you know, reducing it via, like, using a ZK proof or a multi-proof framework is quite good. But then let's actually look at the interoperability of, like, traditional finance today. So some things aren't on-chain whatsoever. And let's, you have a small amount of... So this is kind of on a different scale than the previous one because the proof time is a lot smaller. So you have, you know, the time it takes you to do a trade, maybe a few milliseconds. You have some kind of, like, post-trade accounting where you kind of take everything together. But then you have this T plus one settlement. So T plus one settlement is essentially a proof.
