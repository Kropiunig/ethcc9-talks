# Proving Less, Verifying More: 3.5× Faster L2 Validity Proofs
**Mohsen Ahmadvand )** (Zircuit (Monad) --- Monroe Stage

Hi everyone, my name is Mohsen and I'm gonna talk about faster L2 validity proofs. This is a project that I did together with my colleague Pedro at our time at Zerkit. I'm now with Mona Foundation. So let's get into it and see how we went through this problem. The first thing that I need to establish before talking about the optimization is how does Optimism work when it comes to derivation. So as an L2, in order to be trustworthy for users, you have to submit all the transaction data on L1, so that's your anchor of trust. And once the data is submitted on L1, the rollup has this mechanism that continuously monitors the published data on L1, goes through all the blocks, validates, you know, all the transactions that are relevant for the DA, the call data and blobs, assembles back, you know, those frames and channels, these compressions that Optimism does, and then executes, you know, those recovered blocks from L1. So basically, the entire state of L2 is dependent on what is already shown on L1, and at this stage, you know, the rollup is considered finalized or secure. And against this secure state, you can do your withdrawals. So basically, everything, you know, is anchored on L1. You have a role of a batcher here that submits, you know, your DA, the list of transactions compressed, to an inbox, and that's, you know, the role, that's the actor actively involved in doing that. Now, if you wanna do validity proofs for rollups, the typical way of doing that is to have a range proof. So a set of blocks, you can decide how large that set is gonna be. So it could be 100 blocks. You do a state transition for each of those blocks. You check everything and then you compute a range proof over that one. And you need to pass, you know, the list of transactions as input to a range proof. You don't wanna submit the range proof directly to the chain, because, you know, it's gonna cost a lot for every range that you have to submit a proof. You rather wanna use recursion in your scheme and compress and aggregate all those proofs of your ranges into one aggregate proof that you periodically submit on L1. So this continuity of those ranges is verified by the aggregate proof. And once that proof is, you know, available on L2 output Oracle, then you can basically start withdrawing. The chain is basically 100% finalized. Now, we did the migration to ZKVM.
