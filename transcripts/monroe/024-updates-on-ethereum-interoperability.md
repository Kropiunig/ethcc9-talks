# Updates on Ethereum Interoperability
**Barnabé Monnot** (Ethereum Foundation) --- Monroe Stage

Hello, bonjour à tous. Hi, my name is Barnabé Monot from the Ethereum Foundation, working on improving UX through Interop and excited to share with you some of the updates that we have on that thread of work today. I'm excited to be in Cannes. Thank you for having me. So what is really our goal here? Ethereum has scaled with L2s that has introduced a lot of new use cases, a lot of new users, differentiated applications. All of that is amazing. The cost for it, though, was fragmentation. And fragmentation is just not good UX, generally speaking. Users have to pay costs in money or latency when they want to move cross-chain. Their balances might look very distracting or confusing. And finally, when they move across chain, chains have different lows, I guess. So they might use different gas tokens. And if you don't have the right one, you might find yourself stuck. So there's a bunch of problems. And we set out to think about how we can fix them at a more fundamental layer. And so this is the gist of the strategy that we've been executing since last year, end of August. Our first, let's say, pillar is try to get some short-term wins, fixing cross-chain UX as much as possible, and relying for this on users acquiring liquidity fast. So using intent-based bridges so that users don't have to pay the latency cost. And trying to make the liquidity for these markets as deep as possible and as wide as possible so that the market is open, democratized, and serves new use cases. So that's the first pillar. The second pillar is more fundamental. It's more actual protocol work. It's basically improving the quality and the latency of L1 confirmations. So the L1 finalizes blocks. This is what tells other systems that the L1 data is valid and they can trigger actions based on it. So yeah, really important for the L1 to be fast in that regard. And also important for L2s to settle back to the L1 as frequently as possible because this is what removes friction when you want to move across chain. So the more we do this work in the backend of reducing the latency, having faster finality, lower settlement times, the more we get close to really competitive native message passing. So what's new as of today? I would say that the tools that we've built along with our partners are mostly ready and they are ready to be integrated. And so right now we're really focused actually on going to market with them and supporting the use cases that will adopt them. I would say there are two main products, which I'll present in the first section of the talk. The first one is the open intent framework. That's a modular, secure marketplace for this fast liquidity. So creating a shared market for users to move assets quickly across chain. And the second one is the interop SDK, which is kind of an entry point, a set of libraries and packages that allows apps and wallets to benefit from cross-chain capabilities. So I'll talk about these two and I'll present also like a few more updates on the interop and UX fronts that I think are particularly exciting. So let me start with the open intent framework. So fairly broad collaboration between EF, OpenZeppelin, Wonderland, Arbitrum Foundation, Across, Hyperlane. There's many people who have participated in building this set of tools and this community. And what is it exactly the problem that open intent framework is attempting to solve? So if you want to move assets cross-chain, let's say the basic option moving to an L2 is to use the L2's canonical bridge. So this is what the L2 has to connect to the L1. And this is basically the validating bridge that tells users of the L1 that what happens on the L2 is valid. So that happens either in an optimistic way where you can challenge the L2 state transition if you think it's wrong. That's optimistic. Or it can happen in a pessimistic way where the L2 gives you proof of its validity using zero knowledge technology or snarks. So these validating bridges are the most secure. Basically, the L1 can be completely convinced that what happens on the L2 is valid, but they tend to be quite slow. So in the worst case for optimistic bridges, it takes seven days to do the trip from L1 to L2 to the L1 because you have to wait until the challenge period ends in case somebody challenges the optimistic state transition. In most rollups, optimistic or ZK, it takes many minutes to go from the L1 to the L2, so deposit assets from L1 to L2. And so these latencies are just not really acceptable for many users. And so the solution that has emerged and I think has been quite competitive is to have, instead of a user mechanically moving their funds across these canonical bridges, have a counterparty on the other side that fronts assets to the user. So if I want to move 5 ETH from mainnet to base, I find someone on base who has 5 ETH and who's willing to give them to me very quickly and be repaid eventually. And the Open Market, Open Intents framework is really this idea of trying to create this shared pool of liquidity and standards that makes a single market that's as deep and as liquid and as wide as possible to reduce user costs and user latency and to make this market more accessible. So this is all built on open specifications, standards such as ERC-7683. And we also have like some infrastructure that is just ready to be deployed and used in production. So this infrastructure more specifically, that's basically like a full stack framework built with OpenZeppelin and partners. It's a set of contracts first, so that you can deploy on chain so that you have these endpoints where users can place their funds and counterparties can fulfill orders. There's an aggregator which is off-chain infrastructure, which is basically like a matchmaker. So finding counterparties that are willing to fulfill your orders. And then there's the solver themselves. So that's the counterparty that manages its own inventory of assets and picks up orders when they think that it's profitable for them. So all this code is open source and can be run by chains that want to add cross-chain capabilities. A new thing that I think is particularly exciting is when this counterparty proceeds to make an exchange with a user, an exchange of funds, there are different ways to settle the exchange to make sure that the exchange is fair, that the user indeed received the funds that they were promised and that the counterparty eventually gets repaid. You can use different forms of settlements. Some of them are less secure. Some of them are more secure. But if you want to use, let's say, the one that's as secure as the guarantee that Ethereum gives to its own rollups, what you need to do is send messages across the canonical bridges that tie Ethereum to its L2s. And so we now have, as of a couple of weeks ago, fully audited deployment of the broadcaster, which is a way of sending these messages around. And so you can trustlessly perform this fair exchange between the user and the counterparty. So this eliminates all safety issues with the exchange. So I think this is very cool. And it's also available as an OIF oracle. So that means OIF bridges can just plug into this settlement contract and they get this trustless capability basically out of the box, which is super nice. So a quick call to action. I think we are now ready to integrate. We actually refreshed the website pretty recently, so you can check it out at openintents.xyz. And we're really happy to chat with a few chains, or many, hopefully, who want to, yeah, bolster their interop capabilities. Right, so that was OIF, so a shared marketplace for fast liquidity cross-chain. Now I'll talk about the Interop SDK, which is kind of a counterpart to this market. It's basically a way for apps and wallets to access the shared liquidity and to also benefit from a broader set of cross-chain capabilities. So currently, Interop SDK is essentially a suite of TypeScript packages. The first one is Interop Addresses. This is built on the standards ERC7930 and 7828. So that gives you access to interoperable names like barnabé.eth at base. So these are like chain-specific addresses that are represented in a human-readable format. And so Interop SDK gives you access to tools so that you can decode and encode these addresses. The second package in the Interop SDK is Interop Cross-Chain. So you can think of it as adding some kind of switchboard where you can just change the Interop provider that you use to connect to different networks. And so, yeah, you can maximize your coverage because now you are a bit more provider-independent. You can avoid vendor lock-in as well by just having access to this switchboard. So it's really a little bit like adding some kind of new RPC to your wallet. But instead of an RPC, that's something that requests or transfers liquidity across chain. So this SDK is developed with Wonderland and it's also fully open. So I invite you to take a look. We have a nice website for the Interop addresses. If you are interested in the integration, you can check it out at interopaddress.com. And an update on this is that we now have an on-chain registry. So for chains to register their chain ID or the name that they want to use for these human-readable addresses. The ENS domain on.eth was recently awarded for this purpose by the DAO. Shout out to Unreliable and Wonderland for making this happen. And so, yeah, arbitrum.on.eth, for instance, can be registered. This maps the arbitrum name to some metadata that allows wallets to figure out how they should call addresses on that chain.
