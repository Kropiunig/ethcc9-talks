# Safenet: A transaction security protocol
**Richard Meissner** (Safe Research) --- Monroe Stage

Hey. Hello. Welcome. I hope you're all happy to be here, as happy as I am. I really enjoyed ECC and on a very sunny place near the sea. And it's sadly already the last day, but for most of us, I think the last day is a very interesting day because it's security day. And I'm Richard. I'm part of the SAFE research team. And in the last couple of months, we have been working on Safenet, which we designed as a defense protocol for self-custody. And yeah, I want to give you an update what we have been cooking and want to start out with, why did we even do this? Why do we think it's necessary that we need another protocol for security? And yeah, if we look at the current state, we can really see that the UX in Web3 is like evolving. You can do more stuff, a lot of complex stuff, just by one click. It gets nicely encoded, but it's very hard to understand what is actually happening. We do have quite some checks for this, but like, for example, on the safe wallet UI and most other wallets, you execute your transaction, you get this nice, you're interacting with a verified contract. There are no threats. Only certain balances are changing. But most of these checks only happen on the UI layer. And we had to learn the hard way that the UI layer can be compromised. And if it is compromised, it becomes really hard for the user to really understand what is happening because on most of their wallets, this is what they see. And it's like, hey, please read this and sign this. And hopefully you know what you're doing. And this is really where we think, like with smart accounts, theoretically, we have a tool that allows us to really break into this and provide another layer. And the smart account and the safe smart account says this paradigm called guards, which actually work that you have before and after an execution, an additional check that can check what does a transaction look like? What did the user sign? What state did it change? And this is not only specific to safe accounts. This is actually in most smart accounts then at present where it's called hooks or modules in some others. But this is a very universal tool and it allows us to do nice stuff like timelocks or co-signers. And these are really useful, but not commonly used so far because they're hard to use, right? And if we look at timelocks, for example, kind of relevant when you look at news around crypto in France, sadly, it is this point where they can give you a deterrence against ranch attacks. Because if suddenly an attacker has to wait a week before they can access your funds, the whole game changes how long can they held your hostage. Another thing that we see are co-signers. And co-signers are a tool where you can have a third party that checks and verifies your transaction. And that this doesn't happen only on the UI layer, but really inside the execution layer, inside the guards. The challenge with co-signers is that, like, what is a good co-signer? Like, there are quite some solutions out there. They are some integrated in the safe wallet UI, namely like hypernative, but many of these solutions are very permissioned. So there are centralized tools. You have to apply to them. You have to pay them. Actually, the transparency is very limited. Like, what do they exactly check, right? You can ask them. They will not give you their trade secret. It is their trade secret, right? And also the accountability is there, but it's very off-chain, if at all, right? Like, you kind of have to use a legal contract to sue them and hope that you win. And this is something that is not scalable. And it's also not scalable when we think about, like, hey, we see there are more chains. Chains are spinning up. Chains are spinning down. And all of these are waiting that one of these permissioned tools supports a new chain. And this is not the flexibility that we need for the current ecosystem. And as mentioned before, if you want to set them up, most of them, again, require you to sign some transactions that you don't understand, configure your guard, and it's massively complex. Looking at all of this, we can see that these security limitations can or will become a growth limitation. Because if users don't feel comfortable, if self-custody, like, if there is no trustworthiness and no security that scales, then self-custody will not scale. Users will not use self-custody at scale. And if we don't solve this, this will become a hard blocker for our adoption of self-custody. Yeah. And that's why we started building SafeNet. And that's why we're launching SafeNet in a beta state today. So what is SafeNet? So we designed SafeNet as the last line of defense, right? Like, we, it is a consensus protocol that is on-chain and that gives attestation that can be used in a guard. It is basically decentralizing the co-signer role so that it is not permissioned, it is not accessible just to an exclusive group, but that everybody can use easily the co-signer logic. And that it bridges the gap, that it prevents this gap between intent and execution. I'm not sure if how many of you listened to the talk from Vitalik yesterday. And he was mentioning this quite like AI is omnipresent. AI is a tool that we have to accept that it is there and it can be good and it can be bad. And on the good side, it allows us to build easier transactions and to interconnects faster. But Vitalik at the same time said, right, like for critical actions, the user should be the one that ultimately approves. So we still have this gap between the intent and the execution where SafeNet for us is also a bridge to bring this closer together. So what makes SafeNet different to all existing solutions? And I think when we look back at the earlier checks that I have shown in the screenshot of the SafeWallet UI, it is really this point where we move the security from a UI pinky promise to the protocol level truth. Like it is enforced. There is no way around it. It's not that your UI gets compromised and suddenly that's disabled. No, you put it on the execution layer. And this, furthermore, it is set up in a way that you can easily take it and run it on other chains, across all chains. And it's theoretically even possible to do this independent of EVM chains and scale it across all the different chains that there are. And one of the things that is for us important is that there is accountability built into the protocol. Since we want to strive for democratizing the co-signer role, people have to be held accountable if they don't do their job. And there needs to be a protocol which is also a path towards a sustainable security protocol. So how does it work? And interestingly enough, at ECC I have heard multiple talks, way more than I expected, about Frost. So I might repeat certain things that we have heard so far. But what we have to say, like, at the core we use Frost and Frost is a round-based threshold signature scheme, right? And we see that our validators, so the validators that act and attest transactions, they are basically the set is represented by a Frost public key or a Frost group ID. And we track this on-chain and they together then generate the attestations that you can then, so the attestations are signatures, Frost signatures, and you can use them on-chain to check them in a guard or in a hook, however you want. This, since this is such a, like, this cryptographic property makes it very nice that you can just take the signature and utilize it on any other chain as long as you know the public key of your validator set. And this makes it very scalable. And since we also want that you can rotate your validators over time, we actually use a very common approach that is utilized by many blockchains, where you can basically have epochs in which validators change and you basically create this validator set shape. And each current validator set basically hands off towards a new validator set. This makes it possible that if you go to some other chain, you actually only need to know this initial genesis validator set and with this you can validate any signature across the other chains very permissionlessly. Another very, very nice property of Frost, if it's used with EC keys, is that you can very cheaply verify it on-chain. So far, a lot of the solutions where if you want to use large validator sets have been very expensive, but with Frost you can combine them together in one signature. It's basically as cheap as adding another signature towards your transaction and you get the verification from a whole validator set. So how does it work on the consensus level? So for us, it's important that we want to be able to scale to larger validator sets and that it is a very deterministic consensus. So in this case, there is no ambiguity. It's very clear what is right and what is wrong, right? Like it's like everybody has to do the Caesar moment from the validators. They have to say this transaction is good, there's no security risk, or this transaction we should not attest. And this, because it's so deterministic, we actually get a Byzantine fault tolerance of 33%. So that means up to one third of our validators can behave malicious. They can try to actively work against the protocol and it will still hold and it will not fail the protocol. Even further, you can say that if validators just go down, like they are offline, it's not that they behave malicious.
