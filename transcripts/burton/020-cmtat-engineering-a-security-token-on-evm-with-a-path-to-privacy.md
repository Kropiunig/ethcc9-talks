# CMTAT: Engineering a Security Token on EVM, with a Path to Privacy
**Ryan Sauge** (Taurus) --- Burton Stage

Hello. Okay, perfect. Thank you, everybody, to be here. So I will start my presentation on CMTRT and how to engineer a security token on EVM with a path to privacy. So just to present briefly the project, CMTRT is a security token standard framework with a multi-blockchain support. So we have one implementation for Ethereum and all EVM-based blockchains, one implementation for Tezos, and specifications for Solana. And we have also explored privacy technology with one implementation for Aztec and one for Zama. And it's used by several different companies, so Torus, UBS for a money market fund, Obligate, and so on. And the last year, we also seen a use with Zand for a stablecoin. For the EVM version, this is the main focus of this presentation. It's written in Solidity, built on top of ERC20 and OpenVPN libraries. For the security, major releases are audited by Alboard, which you have one people just here, and ABDK. And for minor releases, we use static analyzer, and recently, we use also high audit engine tools, such as Nonoman, DJHA, and Weka by Yaky. And for the challenge, we have, when we design a security token, we want the token to be used in several different use cases, so equities, debt, stablecoin, several jurisdictions, so not only, for example, in Switzerland, but also Germany, France, Middle East. And so to achieve that, the codebase may be flexible, modular, and easily adaptable. We also want to integrate with the Ethereum ecosystem through different ERC. So for the architecture, functionalities are separated into different modules, so abstract contracts. And from these different modules, we create base contracts and then deployment contracts. So typically, if you want to come and build your own security token, you can just take the module you want and build your own security token. Otherwise, you can also use a deployment contract and directly use it. We also provide some utility contracts, such as the Rule Engine and Rules, to apply more compliant rules, such as, for example, what is the rules. And for the different Ethereum standards, so we have four categories of Ethereum standards. So we have the token standard, then all standards related to tokenization. Then we have standards which are related to proxy and upgradeable support. And finally, we have pure technical standards, so for gasless, cross-chain transfer, and so on. So I don't have the time to explain all of them, but in general, we will try to implement the full standard. Some standards are only available through specific deployment versions. And now we also try to make the token a multi-chain. So when you want to deploy your security token across different blockchains, you have several different challenges. So you have to maintain the same height and properties on each blockchain. You have to ensure a consistent whitelist blacklist across all these blockchains. And you have to enable transferability between these different blockchains, which is the main focus now for CM2T. So for that, we have several key considerations. So you have to select the bridge and integrate it. You have the compatibility between your token and the bridge. And also, you have to configure the bridge and the support. So typically, we have some specific modules inside CM2T, which are specific to this cross-chain technology. So we have typically one module for Optimist standard and all Optimist-based blockchains, such as Union. We have also one module to support changing CCIP. And for Layer 0, we have now external contracts, so two adapters, which can be used with CM2T to enable cross-chain Layer 0. And for the privacy, so now there are a lot of different technologies, so you have to select which technology you want to use. So we have started with Aztec, written in Noir. And this year, we made a version for Zama. And when you define a security token which is private, you have to decide which information is public and which information is private. So typically, the address, the balance, the total supply. For example, for Zama, all the addresses are public, but the balance are private. And depending on what field you want to put to make public, you will leak some information. So typically, the total supply, if it's made public, you may potentially leak some information when you mint a bound token. And finally, you have to add compliance features, so for example, viewing key to allow an external auditor to see the balance, address freezing, and burn functionality, which may be more difficult to implement on a privacy token. And finally, for the integration, so it's not related directly to the token, but it's maybe more complex for each institution to integrate into their systems. You have additional components to manage for Zama, typically you have the KMS and the gateway. And for Aztec, you have all the encrypted private node, the new interface, and the viewing key. And since the on-chain state is private, you need a private key to decrypt the balance and state. So if you have an explorer or an indexer, you need each time the private key just to see the balance. And it's all. So if you are interested, you can go and see our GitHub. All the contracts are open source, so you can use it, you can build your company with that.
