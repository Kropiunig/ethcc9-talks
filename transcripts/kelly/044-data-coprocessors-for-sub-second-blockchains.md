So historically speaking, when block times were a few seconds to a few minutes, blockchains were considered to be a settlement layer. And users worried about self-custody where not your keys, not your crypto was the overarching theme. And what they cared about is that their funds settled and the fact that they were on a network was decentralized enough so they could leave the system. So the walkaway test, as we call it, right? If you wanted to exit the system, you could. So using that in mind, the builders emphasized on consensus mechanisms that were really good. So early on, it was proof of work and then we migrated to proof of stake. And the developers also focused on the ability to audit contracts that were deployed on-chain. So contracts have become the bedrock of how transactions are actually processed in the ecosystem. And as long as there were enough nodes supporting your bytecode, you could say that your transactions would be executed and that you would have a result from that execution. That was the old normal. And that allowed us to create a kind of generalized transaction engine that we could build for EVM. Here, the idea was that you have this piece of software that sits in between the RPC nodes and the customers. And what the software does is it basically makes a JSON-RPC calls to the underlying RPC node and fills a Postgres database. Now this Postgres database has multiple tables. You can have tables for transactions, for logs, for traces, for anything that you cared about as a developer, you could put that in a database and then expect that to be the source of truth. And now that you have this Postgres database, which is a single source of truth, you can wrap it with any kind of library, maybe a Django backend or even Java APIs, which could read, update, delete any sort of value. Mostly read in this case, because once it's written, it's final, right? And so this is easy to scale. And we could scale this for over a hundred chains. Most of them were EVM chains because the generalized transaction engine had the same JSON-RPC interface. And so that standard allows indexers to scale quite easily. But what we're starting to see now is with chains that are super fast, especially with like lower block times, say close to like a few hundred milliseconds or 200 milliseconds or even less, right? There are some chains that do a thousand transactions per second. Now, how do you index that in this world? That's completely not possible because the size of the underlying tables keeps increasing and the cost of storage rises asymptotically with the size of the database. So the Postgres database here that I've posted is for three different chains, base, Ethereum, and BSC mainnet. And just the logs table, the transactions table, and say the contract balances table is a few terabytes. Now imagine doing this for 50 chains. This easily goes to a hundred terabytes. And imagine having to serve that on SSD-based drive where you have a piece of software running on top. It optimizes for query time where you have to get the data in the shortest amount of time possible, but it also has to be accurate. Furthermore, if you have to do any kind of analytical queries, it becomes even more difficult because now you have to aggregate amongst multiple tables and you have to make the joins between tables by using foreign keys. Right? So in the world where chains are like moving super fast and blocks are finalizing within a hundred, 200 milliseconds, this just simply stops working, right? So initially, maybe about three years ago, before chains became this fast, because the size of the database was growing, we thought one way to kind of mitigate that problem would be to distribute the workload. It's obvious. So we came up with the notion of a distributed ETL pipeline using the Ethereum Wayback Machine. And the idea is pretty simple. You have a network of validators. Each validator is assigned a specific role in the pipeline, whether it's extraction, transformation, or load, right? And using crypto economic incentives, using an underlying token, we could get them to coordinate on the different parts of the ETL pipeline. But obviously this has certain limitations too, because the most important part of the pipeline is extraction, right? And so all the assets converge towards the producers of the underlying artifact. And the network became a bit imbalanced. And so the incentives were progressing towards concentration of capital towards the extraction part alone. And so the other parts of the network, especially close to the application layer, for example, for the query node operators, for the applications, decreased. And so we had to come up with an understanding of how to suppress this kind of artificial concentration of capital for validators. And so we have to again, think about the token economics, about how we distribute the token and the valuation for the different parts of this extraction pipeline. And basically come up with a verification mechanism, which is lightweight. So in the current sort of conception of how blockchains have evolved, because of the lower block times, what we see is the notion of the settlement layer has kind of shifted. It's gone from being a place where you would put a transaction and expect it to be completed to a world where it's a bulletin board, where a piece of information exists for a very short period of time and you can verify that it's true, but nobody needs to keep a hold of it. And there's this EIP called EIP-44 that actually plans to expire all the historical state of Ethereum and move towards a more stateless world where clients can specify what's the last block that they want to sync from, what's the first block and what's the last block that they want to sync from. So not all RPC nodes then have the historical state of the chain. So this is the problem that we are facing now is with all these different changes, the users want faster trades. They want responsive balance updates on their applications. They still want the exact data and they want to be able to access it at any given time. So we have to fulfill those requirements while builders are constantly pushing on the number of transactions that they want to do by having bigger blob sizes, by decreasing the fees of the transactions and having faster finality. So this world is already here. That's what I'm trying to say is we are doing 50,000 transactions per second on chains like HyperCore and Solana and even MegaEats. And the primary receivers of this data has shifted. Early on it was human beings who were looking at their transactions going like, okay, this takes about 12 seconds. I'll wait for it to complete. But now it's agents. Agents don't require a human in the loop. They want to be able to make decisions on new data that's coming in and they want to be able to make it fast. So the latency requirements have drastically moved from like a few seconds to a few milliseconds. And imagine that within a blink of an eye, which is 150 milliseconds, a hundred transactions have occurred on chain. So it's literally not possible for a human being to catch up at this rate. And what this leads to is a difference between what's at the tip of the chain and what the agents know from RPC providers or from data indexers. And this gap is growing.


As more blocks are being added, the gap between the tip of the chain and what the agents can receive from the underlying indexer nodes is growing further and further, and this leads to what we call state blindness. The applications at the front essentially don't know what the actual state is because they're so far back from what they can actually receive. And this is the example that I wanted to post, is we have to rethink the arch. So Lefris is the founder of Rocki app, which is a self-custodial portfolio management app, and what he's trying to say here is the indexer situation for EVMs is extremely bad. So you don't want to be in this world where you have super fast transactions and you have finality and security, but you don't actually know what's on chain. And this is where we have to move our thinking towards block stream processing and the medallion architecture. So let's get into that. The easy way to get out of this problem is to actually go back to the source. So what's the source of the data? It's the actual RPC node, right? At the time that an RPC node is syncing, you can run an engine that exports the entire block along with the traces of the block in a protobuf binary format, right? And you can do this by scaling workers. You can have, for example, if you wanted to index 10 million blocks, you can have 10 different workers at different block segments, say block 0 to block 1 million, then one from 1 million to 2 million, etc. And now you can concurrently index 10 million blocks by these 10 different workers and then linearize it at the time that it's required at the end of the pipeline and push it to a message event system. So in this case, we are using Kafka for the underlying block stream processing. So now that you can use this underlying block stream processing, you can scale it for multiple chains. So you can have one chain produce into its own separate topic, and you can have multiple chains doing this into the Kafka cluster. And then independently, you can decide which part of that data is the lowest latency, high throughput data that you might need. So we get a lot of storage savings as a result of what we call tiered storage. The idea here is very similar to like expiring data, but still keeping it in the sense that remote long-term storage holds data from like Genesis to the last year. And from the last year to like the month and last few weeks, you keep that on a hot SSD, right? So you can still have super fast access writes and reads from the SSD. While if you want to make a point query to historical data, you can still use the long-term storage. And this allows us to reduce the cost of keeping this data for a long time and also have multiple consumers that according to the needs of the consumers, can point towards that part of the storage. And this unlocks a lot of more different applications. So now that you have all the data in Kafka clusters, which are partitioned by topics per chain, you can have different consumers consume it from different offsets. So depending on how much data you need to do your analysis, whether it's just the five minutes of data from this day or maybe one year's worth of data, you can point the offset of your consumer, right, to that particular point and start consuming from there. And this essentially allows you to also have connectors for clients where a connector is something that consumes from a given topic. And here the topic can be, say, transfers for a given chain. And the sync can be like another user queue or it can be their own database or it can be their own data warehouse. So this allows us to like decouple the production of the data from the consumption of the data. And then the consumers can define what kind of a format they need their data in instead of it having to be a Postgres database where the underlying schema is predefined by us. This allows us a lot of convenience in spinning up jobs ephemerally, giving you the data you want and then delivering the results. Right. And this decoupling of producer-consumer architectures allows you to have this notion of a medallion architecture where the medallion essentially is divided as bronze, silver, and gold, where bronze is that raw block information with all the traces underneath. And then you have a process in the middle that takes, reads from this topic and produces data into a different topic where it can have enrichments. Those enrichments can be ABI decoding, token metadata, some sort of annotation like USD pricing of the token, etc. And this allows you to compress the information from just raw data, which is like for base, it's around 90 terabytes, to, for example, transfers, which is only 1.2 terabytes. So you already get about 10x savings in terms of space. Right. And then you can take the silver data and further refine it where you have another process that can extract from transfers what are the DEX swaps that are occurring, what are the OHLCV candles, so open, high, low, close, the different volumes that you care about. So you can derive actionable information from the gold data, and then you can point your consumer to whichever piece of information you need at the lowest latency with the highest throughput. So here I have some real-world data compression use cases where we have gone from bronze data, which is a few terabytes, to actual actionable data in a few gigabytes. Right. And we can do this pretty quickly. So I have a table here on the right, which has live metrics from this architecture where end-to-end latency going from bronze data to gold data is a few seconds, while the actual processing time of the block is a few milliseconds. So this gives us both high storage savings as well as time savings. Instead of waiting for the indexer to index everything completely, you can just choose what amount of data you want and just focus on it using this producer-consumer decoupling. And the primary use cases for this data that we are seeing is in DeFi, where you have these real-time agents who are processing OHLCV metrics or doing DEX arbitrage or doing some sort of liquidation monitoring. And the other examples here are also stablecoin monitoring and actual analytics, where you want to aggregate a lot of data and derive some insights from it. So yeah, that's basically the presentation. And here I have a single slide that kind of sums up everything that I talked about, which is we as a community are facing the challenge of state blindness as a result of block space becoming a commodity. And sub-second blockchains exist. Blocks finalize in a few milliseconds, but traditional indexers are not able to keep up. And so people are stopping to use applications on those chains.
